# AGENTS.md

## 1. 当前项目阶段

本项目当前处于药品批发 ERP 验证型 Demo 阶段。

当前目标是：

> 实现全国药品批发核心流程，并通过广东增强 Profile 展示省级追溯和监管接口的模拟能力。

本项目不是：

* 正式商业 ERP；
* 已通过 GSP 检查的系统；
* 已完成计算机系统验证的系统；
* 已覆盖全国所有省级规则的系统；
* 可以直接承载真实药品经营数据的生产系统。

Demo 验证通过后，允许后续团队重新进行商业系统需求分析、架构设计和代码重写。


## 5. Demo 里程碑

### M2：广东增强与质量展示

* 质量事件和库存冻结；

## 8. 核心业务硬规则

以下规则不得为了快速演示而绕过：

1. 供应商首营企业审核通过后才能采购；
2. 首营品种审核通过后才能采购；
3. 客户资质有效且经营范围匹配后才能销售；
4. 采购订单不能直接增加可销售库存；
5. 收货和验收必须分离；
6. 验收应逐批执行；
7. 验收合格后才能形成合格批号库存；
8. 库存必须按药品、批号、有效期、仓库、库位和质量状态管理；
9. 每次库存数量变化必须生成 `stock_movement`；
10. 状态变化必须留下质量状态变更和审计记录；
11. 待验、冻结、不合格和过期库存不得销售和出库；
12. 销售出库必须经过出库复核；
13. 批号必须能够追溯到采购、收货、验收、库存、销售、出库和客户；
14. 关键数据不得无痕修改；
15. 审计记录、库存流水、审批记录和关键业务记录不得随意删除。

## 9. 省级能力规则

全国核心业务不得依赖具体省份。

广东能力通过：

```text
Profile
→ Rule Pack
→ Adapter
```

实现。

禁止在业务代码中散落：

```java
if (province == "GD") {
    // 广东逻辑
}
```

首版不得建设：

* 通用规则引擎；
* 可视化规则编辑器；
* 动态脚本平台；
* 自定义表达式语言；
* 面向任意省份的通用政策配置平台。

首版 `Rule Pack` 只能由少量明确配置和策略类组成。

外部监管和追溯接口必须通过Adapter隔离，并明确标识为Mock。

## 13. 数据库规则

数据库名称：

```text
pharma_erp
```

规则：

1. 表名和字段名使用小写下划线；
2. Java属性使用小驼峰；
3. 金额使用`BigDecimal`和`decimal`；
4. 日期使用`LocalDate`或`LocalDateTime`；
5. 业务状态使用枚举；
6. 禁止魔法字符串；
7. 关键业务表原则上不物理删除；
8. 数据库变更使用明确的迁移脚本；
9. 已执行的迁移脚本不得直接改写；
10. 禁止直接修改库存总数量而不生成流水。

统一使用：

```text
stock_batch
stock_movement
```

不得新建同义的：

```text
stock_transaction
```

## 14. 后端分层规则

Controller：

* 接收HTTP请求；
* 参数校验；
* 调用Service；
* 返回统一结果；
* 不写复杂业务逻辑。

Service：

* 业务规则；
* 状态流转；
* 权限以外的业务校验；
* 事务控制；
* Mapper调用。

Mapper：

* 数据访问；
* 简单CRUD优先使用MyBatis-Plus；
* 不为简单查询建立复杂XML。

Entity：

* 对应数据库表。

DTO：

* 接收请求参数。

VO：

* 返回展示数据。

Enum：

* 表示受控业务状态。
