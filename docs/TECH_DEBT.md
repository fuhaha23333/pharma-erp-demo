# 技术债基线

## 1. 文档说明

本文记录 Goal 1.1 审计时已经由仓库或命令证据发现的技术债、风险和待确认事项。技术债不等于“代码一定有 bug”，而是当前实现与后续可验证、可维护、可扩展目标之间的差距。

- 审计日期：2026-07-18
- 基线分支：`main`
- 基线提交：`cad7ff6 docs: add Goal 1 documentation placeholders`
- 事实基线：`docs/CURRENT_STATE.md`
- 本 Goal 只记录问题和建议方向，不修改业务代码、配置或数据库。
- 涉及药品经营、GSP 或监管解释但缺少正式依据的事项统一标记为“合规待核验”，不作为法律结论。

## 2. 严重程度定义

| 级别 | 含义 |
| --- | --- |
| P0 | 继续开发前必须处理，否则存在严重数据、安全或项目失控风险。 |
| P1 | 进入下一主要业务模块前应处理。 |
| P2 | 当前可以暂时保留，但必须进入明确计划。 |
| P3 | 不阻塞当前阶段的优化项。 |

本次共记录 22 项：P0 0 项、P1 12 项、P2 10 项、P3 0 项。优先级是基于当前 Demo/MVP 阶段的工程判断，后续若获得新的业务、部署或法规证据，应重新评估。

### 2.1 阶段性阻断规则

当前 P1/P2 分级建立在项目仍是**未投入生产、未导入真实经营数据的开发原型**这一前提上。某项技术债在当前开发阶段被列为 P1 或 P2，不代表它可以被带入试用、外部访问或真实业务环境。

在以下任一情况发生前：

- 导入真实经营数据；
- 企业用户试用；
- 部署到外部可访问环境；
- 实际承载药品经营业务；

以下事项必须作为阻断项完成并验证：

- 环境和测试数据源隔离；
- 登录和接口权限；
- 操作人、操作日志及修改前后值；
- 数据库迁移管理；
- 数据备份和恢复；
- 其他直接影响数据真实性、完整性和追溯性的事项。

未完成并验证上述阻断项前，不得将系统认定为可试运行或可商用。

| 分类 | 安全、配置 |
| 严重程度 | P1 |
| 客观证据 | `.gitignore` 忽略 `application-local.yml`；审计确认 `backend/src/main/resources/application-local.yml` 的密码字段非空，且该文件与 `backend/target/classes/application-local.yml` 一致；Maven compile/test 均报告复制 2 个主资源。Maven 复制 `src/main/resources` 属于正常构建行为；本次只确认该本地配置已进入 `target/classes`，未验证最终 JAR 内容。密码具体值未在文档或命令输出中回显、记录，也未确认该值是否为仍然有效的真实凭据。 |

## 4. 处理原则

1. 在进入新的业务模块前处理 TD-001；若确认凭据真实有效或制品已经外发，立即按 P0 安全事件处理。
2. P1 项不是要求一次性大重构，应按 Goal 拆成小任务，每次明确范围、测试和回滚方式。
3. 所有“合规待核验”项必须先取得正式业务和法规依据，再转化为数据库、接口和测试规则。
4. 标记为“推断”的问题必须先通过日志、测试或运行证据验证，不能直接按已确认 bug 修复。
5. 技术债处理完成后应更新本文件的状态或形成可追踪的后续记录，避免只修代码不更新基线。
